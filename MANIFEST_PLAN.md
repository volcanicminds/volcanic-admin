# Manifest — Implementation plan (multi-repo)

> Operational plan for the **manifest capability** of the Volcanic Minds ecosystem: the backend
> (`@volcanicminds/backend`) generates OOTB a **descriptor manifest** (core + custom project endpoints);
> the admin (`@volcanicminds/admin`) consumes it at **build-time** and layers customizations on top; Dionisi
> is the first real instance that validates it.
>
> **Status**: design in progress. Target contract = **Manifest v2** (see `MANIFEST_DESIGN.md`). The admin engine
> today implements **v1** (`src/engine/types/manifest.ts`): v2 is a planned breaking refactor and
> migrated in tasks ADM-1/M3.
>
> **Repos involved**: `volcanic-backend` (framework), `volcanic-backend-sample`, `volcanic-admin` (admin product),
> `dionisi-group/backend` (BE consumer), `dionisi-group/backoffice` (admin instance).
>
> **Twin document**: `MANIFEST_DESIGN.md` (M0-1) — contract and lifecycle. All manifest documents
> live in this repo (`volcanic-admin`).

## Fixed architectural decisions

1. **One source, two projections.** `route + schema (+ entity)` feed *both* the OAS/Swagger (already OOTB) *and*
   the manifest. Consistency by-construction, zero double maintenance.
2. **Canonical field identity = `(resource, field)`**, not `(schema, property)`. The generator **collapses** the N
   schema projections (body/response/public) onto the canonical identity → per-entity override, not per-schema.
3. **BE ↔ Admin split**:
   - **BE** = structure + domain + security/federation: exposed resources/fields (whitelist/blacklist), domain-types,
     enum-values, relations, constraints (required/min/max/pattern), roles, real endpoint binding (CRUD + actions).
   - **Admin** = presentation: widgets/components, sidebar groups (label/icon/order), `form.group`/sections, layouts,
     colors, i18n labels, views, UI defaults, theming, dashboard, shortcuts.
4. **3-level model**:
   - **L0 zero-config** (BE, heuristics): group=dir, titleField=heuristic, types/required from schema.
   - **L1 structural hints** (BE, inside `config` of `routes.ts`): `group`, `resource:{titleField,subtitleField,globalSearch}`.
     Domain, optional, additive, no UI.
   - **L2 presentation** (Admin, overrides): all the UI.
5. **Build-time consumption** with **`generated` / `overrides` split**:
   - `manifest.generated.ts` → **always** produced by the BE, never hand-edited (overwritten on every refresh).
   - `manifest.overrides.ts` → project-only, empty scaffold on first generation, never touched by the regenerator.
   - admin at build: `merge(generated, overrides)`. Resolves drift while preserving the overrides.
6. **Security**: the build-time manifest is **full** (not per-user) → it carries the `roles[]` declared per capability; per-role gating
   is admin at runtime + **authoritative enforcement on the BE endpoints**.
7. **AutoCrud NOT implemented now** (neither framework nor Dionisi): Dionisi has rich hand-written endpoints; the manifest
   works on top of them. It remains a separate future capability.

### Decisions locked in the brainstorming (2026-06-28)

8. **Contract = Manifest v2**, unified array **`capabilities`** (`CapabilitySpec[]`): collapses the v1
   `permissions` + `capabilities`(boolean) + `actions` into **a single list**, with `roles` in one place only.
   Same type used in `resources[].capabilities` (CRUD + actions on the resource) and in the **top-level `capabilities[]`**
   (standalone *operation* sections, not tied to an entity). `version: 2`. `search` remains list config
   (`resource.search`), NOT a capability. Detail in `MANIFEST_DESIGN.md`.
9. **Scope = resources + operation sections**: the manifest has `resources[]` **and** top-level `capabilities[]` for the
   non-CRUD endpoints (storefront/tool/pure actions → dedicated page/action generated by the engine).
10. **Fidelity = JSON Schema only**: the generator does NOT depend on the data layer. Consequence: `relation` comes out **thin**
    (only the target `resource`; no `kind`/`foreignKey`, not deducible from the schema); enum-values only if present
    in the schema. The rest is filled in the admin `overrides`.
11. **Sensitive = graduated policy**: `password` excluded from read/list but allowed **write-only** in create/update;
    `token`/`mfaSecret`/`externalId` excluded **always**. Blacklist extensible via config. No binary exclusion.
12. **Naming/coverage**: omni → **`globalSearch`**; `image` config → not BE (admin override); `defaults`
    (pageSizes/bulk) → admin only; `list.visible/sortable` → admin (BE can impose `sortable:false`); `titleField`/
    `subtitleField` → BE indicates the field list, the admin optionally an i18n template with variables, fallback =
    concatenation.
